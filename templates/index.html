📖 Release Readiness — Complete Guide

Release Readiness is an AI-powered release coordination dashboard that replaces manual spreadsheets and Teams messages. Developers nominate services for each weekly release, AI validates health and generates release notes enriched with Jira ticket data, and deployments to UAT are triggered directly via GitHub Actions — all from a single dashboard.

🔄 Typical Weekly Release Workflow

1
Monday–Wednesday: Developers deploy code to UAT using the 🚀 Deploy tab (GitHub Actions) and test their changes.
2
Nominate Services: Once verified in UAT, developers nominate their service on the 📋 Board tab — version is auto-captured from the live cluster. Add Jira IDs for the changes being released.
3
QA Validation: Run 🔀 Version Drift to check for mismatches, run 🤖 AI Readiness for health checks. Use 📜 Version History to rollback if needed.
4
Wednesday 5 PM — Cutoff: Release Manager clicks 🔒 Lock Board. No more regular nominations accepted. ⚠️ Exception Nominations are still possible with approver and reason (for Thursday hotfixes).
5
Generate Artifacts: Go to 📦 Export tab → 🤖 AI Release Notes (fetches Jira ticket summaries via MCP). Export JSON/YAML manifests for production deploy pipelines.
6
Friday — Release Day: Deploy to production using exported manifests. Click ✅ Mark Released to archive the board. A new weekly cycle begins automatically.

🏁 Getting Started

🔐 Step 1: GitHub Login

Click "Login with GitHub" in the sidebar to authenticate. This enables deployment workflows and associates your identity with board actions.

Authentication modes:
  • OAuth 2.0 (recommended) — Secure browser-based login. Requires GITHUB_CLIENT_ID + GITHUB_CLIENT_SECRET.
  • PAT mode — Shared Personal Access Token for service accounts. Set GITHUB_TOKEN env var.
  • Supports GitHub Enterprise — set GITHUB_URL to your GHE instance.
Why it's needed:
  • Your GitHub identity is logged in the Audit Trail for every action
  • Required for the 🚀 Deploy tab (triggers GitHub Actions using your token)
  • Board nominations and AI features work without login — only Deploy requires auth

📋 Board & Nominations

📋 Release Board

The Board is the central hub for every release. It shows all services nominated for the upcoming Friday release with their locked versions.

How to nominate a service:
  1. Choose ● K8s Service or ⚡ Custom Component in the sidebar
  2. Select a service from the dropdown — Image Tag and Helm Chart are auto-filled from the live UAT cluster
  3. Enter your name in Your Name
  4. Enter associated Jira IDs (comma-separated, e.g., BILL-101, BILL-102)
  5. Add optional Release Notes describing the change
  6. Click 🚀 Nominate
Key behaviors:
  • Version is locked at nomination time — if someone deploys a newer version to UAT later, the board keeps the original nominated version
  • To update, re-nominate the same service (previous version is saved in history)
  • Toggle between Tile view (▦) and List view (☰)
  • Each card shows: version tag, Helm chart, readiness badge, 🎫 Jira badges, nominator, date, and notes
  • Click 👁 Details to see full service info, 📜 History for version timeline, ✕ Remove to withdraw
Board states:
  • ● Open — Accepting nominations (Monday through Wednesday cutoff)
  • 🔒 Locked — Past cutoff. Regular nominations blocked. Exception nominations still allowed with reason + approver.
  • ✅ Released — Archived as read-only. A new cycle begins.
💡 Tip: The summary bar at the top shows nominated count, readiness breakdown (Ready / Review / Risk), release date, and cutoff time at a glance.

⚡ Custom Components (Non-K8s)

Not everything runs in Kubernetes. Custom Components let you nominate Spark jobs, PySpark pipelines, mobile apps, config rules, or any non-K8s artifact.

How it differs:
  • Click ⚡ Custom Component tab in the nomination form
  • Version is manually entered (not auto-detected from cluster)
  • Component types and descriptions are configured via CUSTOM_COMPONENTS env var
  • Jira IDs work the same — link Jira tickets to custom component releases
💡 Config example: CUSTOM_COMPONENTS=ingestion-pipeline:Spark:Main data ingestion,etl-transformer:PySpark:Data transformation

📜 Version History & Rollback

Every time a service is nominated or re-nominated, the previous version is saved in a timeline. Click the 📜 History button on any service card to view and rollback.

What you can do:
  • View full timeline — who nominated which version and when
  • Click "↩ Rollback to this version" to instantly revert to any previous nomination
  • All rollbacks are logged in the Audit Trail with from → to version
💡 Use case: You nominated billing-service v2.3.5, but QA found a regression. Rollback to v2.3.3 without redeploying — the board nomination reverts instantly.

⚠️ Exception Nominations (Post-Cutoff)

After the board is locked (Wednesday 5 PM cutoff), developers can still nominate services as exceptions. This is for Thursday hotfixes or critical changes that must go into the Friday release. Every exception is tracked and reported for weekly review.

How it works:
  1. Try to nominate a service after the board is locked
  2. An ⚠️ Exception Nomination modal appears automatically
  3. Enter a Reason (why this is needed after cutoff) — required
  4. Enter an Approver (who approved this exception) — required
  5. Click Submit Exception Nomination
What gets tracked:
  • Board cards — Exception services show an ⚠️ EXCEPTION badge with approver and reason
  • Sidebar — Exception count displayed in Quick Info when > 0
  • Audit Trail — Logged as exception-nominate or exception-re-nominate with amber highlighting
  • AI Release Notes — A ⚠️ Post-Cutoff Changes section is added, flagging these as higher-risk items
  • Exception Report — GET /api/release/exceptions returns stats grouped by requester and approver
💡 Why it matters: Exception tracking creates accountability. Weekly reports show how many post-cutoff changes are happening and who is requesting them — helping the team reduce late changes over time.

🎫 Jira Integration

🎫 Linking Jira Tickets to Releases

Each nominated service can have Jira ticket IDs attached, connecting the release board to your team's issue tracker. This enables AI-powered release notes that summarize what actually changed based on Jira descriptions.

How to add Jira IDs:
  1. When nominating a service, enter Jira ticket IDs in the Jira IDs field
  2. Use comma separation for multiple tickets: BILL-101, BILL-102, BILL-103
  3. IDs must follow the standard format: PROJECT-NUMBER (e.g., DEVOPS-123)
  4. Jira IDs can be updated by re-nominating the service
Where Jira IDs appear:
  • Board tile view — Styled 🎫 badges on each service card
  • Board list view — Dedicated "Jira" column in the table
  • Service detail modal — Full Jira ID list when you click 👁 Details
  • AI Release Notes — "Jira Tickets" column + "What's Changed" section
  • AI Chatbot — Ask "What Jira tickets are in this release?" or about a specific service

🔌 Jira MCP Server Connection

Release Readiness connects to your organization's Jira MCP (Model Context Protocol) server to fetch ticket details (summary, description, status, type, priority). This data powers the AI release notes generation.

How it works:
  • When you click 🤖 AI Release Notes, the system collects all Jira IDs from nominated services
  • For each Jira ID, it calls the MCP server's jira_get_issue tool via JSON-RPC over HTTP
  • Ticket summaries and descriptions are injected into the Gemini prompt
  • Gemini generates release notes organized by change type: 🆕 Features, 🐛 Bug Fixes, ⚡ Improvements, 🔧 Maintenance
  • Results are cached for 5 minutes to avoid redundant API calls
Configuration:
JIRA_MCP_URL=https://jira-mcp.your-org.com
JIRA_SERVICE_ACCOUNT=jira-svc-account
JIRA_PAT_TOKEN=<your-jira-pat-token>
💡 Graceful degradation: If no Jira MCP server is configured, or it's unreachable, release notes still generate — just without Jira descriptions. No errors, no breakage.

✅ Quality & Validation

🔀 Version Drift Detection

Compares the nominated version on the board against what's actually running in UAT right now. Catches cases where someone deployed a newer version after nomination.

Drift levels:
  • 🟢 No Drift — Nominated version matches what's live in UAT
  • 🟡 Minor Drift — UAT has a newer patch/minor version. Developer should re-nominate with the correct version.
  • 🔴 Major Drift — Different major version in UAT. Needs immediate investigation — possible unintended deployment.
💡 When to run: Run drift checks before cutoff (Wednesday) and after any UAT deployment to ensure the board reflects reality.

🤖 AI Readiness Check

Powered by Google Gemini. Performs automated health analysis on every nominated service by examining live Kubernetes cluster data (pods, events, resources, probes).

What AI checks:
  • Pod health — replica count, crash loops, restart count in last 48h
  • K8s events — warnings, errors, scheduling failures, ImagePullBackOff
  • Resources — CPU/memory requests vs limits, potential OOM risk
  • Probes — liveness/readiness probe configuration and failure rates
  • Image tag risk — flags :latest tags as unpinned supply chain risk
  • Version change risk — major version bumps flagged for extra review
Readiness scores:
  • 🟢 Ready (70–100) — Service is healthy. Safe to include in release.
  • 🟡 Review (40–69) — Some concerns found. Release Manager should investigate before approving.
  • 🔴 Risk (0–39) — Critical issues. Should not be released until resolved.
💡 Tip: Readiness results are shown as badges on each board card and in the summary bar at the top. Click 👁 Details on any service to see the full AI analysis.

📜 Audit Trail

Every action on the release board is recorded with timestamps, who did it, and what changed. Full traceability for compliance and post-mortems.

Events logged:
  • nominate — Service added to the board with version and Jira IDs
  • re-nominate — Version updated (from → to) with reason
  • exception-nominate — Post-cutoff nomination with approver and reason (amber highlight)
  • exception-re-nominate — Post-cutoff re-nomination with approver and reason
  • rollback — Version reverted via Version History
  • remove — Service withdrawn from the release
  • deploy_triggered — GitHub Actions workflow dispatched (includes run ID, who triggered, inputs)
  • finalize — Board locked at cutoff by Release Manager
  • release — Board marked as released (archived)

📦 Export & Release Notes

📦 Export Manifests

Download the release manifest in JSON or YAML format. These machine-readable manifests can be fed into Helm, ArgoCD, or custom deployment pipelines for the production release.

Manifest contents:
  • Release name, date, status, cutoff timestamp
  • Per-service: name, image, image_tag, helm_chart, nominated_by, readiness, notes

🤖 AI Release Notes (Jira-Enriched)

Generates professional, human-readable release notes powered by Google Gemini. When services have Jira IDs, the system fetches ticket details via the Jira MCP server and weaves them into the notes.

What gets generated:
  1. Executive Summary — 2-3 sentence overview of the release
  2. Service Table — Service | Version | Jira Tickets | Change Summary | Risk Level
  3. What's Changed — Changes organized by type:
    • 🆕 New Features — from Story/Feature Jira tickets
    • 🐛 Bug Fixes — from Bug tickets
    • ⚡ Improvements — from Improvement tickets
    • 🔧 Maintenance — from Task tickets
  4. AI Risk Assessment — Overall release risk based on readiness scores and change scope
Output options:
  • 📋 Copy Markdown — paste directly into Jira, Confluence, or Teams
  • 📄 Copy as HTML — paste into email clients
✨ Gemini Powered badge appears when notes are generated by AI. 📄 Template badge appears when Gemini is not configured (deterministic fallback). Both modes include Jira data when available.

🔒 Board Lifecycle Controls

  • 🔒 Lock Board (Cutoff) — Freezes nominations. Typically done Wednesday 5 PM. Only Release Manager should use this. All nomination buttons are disabled after locking.
  • ✅ Mark Released — Archives the board as read-only. The release date advances to the next Friday and a fresh board is created for the new cycle.

🖥️ UAT Environment & Deployment

🖥️ UAT Environment

A live view of all services currently deployed in the UAT Kubernetes namespace. Independent of the nomination board — shows what's actually running right now.

What it shows:
  • Service name, kind (Deployment / StatefulSet), replica status (ready/desired)
  • Current image tag and Helm chart version from the cluster
  • Availability status — whether all pods are ready
  • Use this to verify what's running vs what's nominated on the board
  • Click 🔄 Refresh to get the latest data from the cluster

🚀 Deploy (GitHub Actions)

Trigger GitHub Actions workflows directly from the dashboard. Connects to your org's deployment repository and auto-discovers all available workflows.

How it works:
  1. Login with GitHub first (OAuth required for API access)
  2. Workflows are auto-discovered from the repo — just add workflow YAML files and they appear
  3. Each workflow card shows: name, file, last run result (✅/❌), duration, who ran it
  4. Click a workflow to expand — dispatch inputs are parsed from the YAML and rendered as form fields
  5. Fill in inputs (service name, version, namespace) and click ▶️ Run Workflow
  6. Real-time status polling shows step-by-step progress with live updates
Safety features:
  • environment input is 🔒 forced to UAT — staging/production cannot be targeted from this tool
  • Every deploy trigger is logged in the Audit Trail with GitHub user, inputs, and run ID
  • Deploy history shows all past runs with status and links to GitHub
Configuration:
DEPLOY_REPO=your-org/app-deployments
GITHUB_CLIENT_ID=<oauth-client-id>
GITHUB_CLIENT_SECRET=<oauth-secret>
BASE_URL=https://release-readiness.your-domain.com
💡 Tip: Workflows need a workflow_dispatch trigger with inputs. choice inputs render as dropdowns, string inputs as text fields.

💬 Conversational Release Assistant

A natural language AI assistant powered by Google Gemini with function calling. Ask questions about the release in plain English and get instant, data-backed answers with formatted tables and status reports.

Example questions:
  • "What's in this Friday's release?" — Full board summary with versions, Jira tickets
  • "Is billing-service nominated?" — Specific service status with all metadata
  • "Check for version drift" — Compares board vs live UAT
  • "Show readiness scores" — AI health check summary for all services
  • "Who nominated auth-service?" — Nomination details and history
  • "Show the audit trail" — Recent board activity
  • "What's running in UAT?" — Live cluster overview
  • "What Jira tickets are in this release?" — All Jira IDs across nominated services
Features:
  • Multi-turn conversations — remembers context within a session
  • Function calling — Gemini calls 6 release-specific tools to query live data
  • Quick-ask pills — pre-built suggestion buttons for common queries
  • Markdown rendering — tables, bold, code blocks, lists
  • 🗑 Clear button — resets conversation history for a fresh start
  • Read-only — the chatbot can query data but cannot modify the board
💡 Powered by: Gemini function-calling with 6 tools — release_get_board, release_get_service_status, release_get_drift, release_get_readiness, release_get_audit, release_get_uat_services. Requires GCP_PROJECT_ID or GEMINI_API_KEY.

❓ Frequently Asked Questions

Q: What happens if I nominate the same service twice?
The old version is saved in Version History and the board updates to the new version. Both nominations are logged in the Audit Trail.
Q: Can I add Jira IDs after initial nomination?
Yes — re-nominate the service with the same version and add the Jira IDs. The field merges with existing IDs.
Q: What if Jira MCP server is not configured?
Everything works normally. Jira IDs are still displayed on the board, but the AI Release Notes won't include Jira ticket descriptions — they'll use service metadata and notes instead.
Q: Can I deploy to production from this tool?
No. The environment input is locked to UAT. Production deployments must go through your standard CI/CD pipeline using the exported manifests.
Q: How does the board persist data?
Board state is stored in a Kubernetes ConfigMap in the same namespace. If the service account lacks ConfigMap write permissions, data falls back to a file on a PersistentVolume. Both methods are automatic — no configuration needed. In mock mode, data is in-memory (resets on restart).
Q: Can I nominate a service after the board is locked?
Yes — as an Exception Nomination. You'll need to provide a reason and an approver name. The service will be flagged with an ⚠️ badge and included in the weekly exception report.
Q: Where can I see how many exceptions happened this week?
The sidebar shows exception count when > 0. For detailed stats, the Release Manager can call GET /api/release/exceptions which returns counts grouped by requester and approver.
Q: What if Gemini is not configured?
AI Readiness and AI Chatbot are disabled. Release Notes fall back to a deterministic template (still includes Jira IDs if provided). The board and all other features work without AI.
