📖 Release Readiness — Complete Guide

Release Readiness is an AI-powered release coordination dashboard that replaces manual spreadsheets and Teams messages. 🔄 Typical Weekly Release Workflow

1
Monday–Wednesday: Developers deploy code to UAT using the 🚀 Deploy tab (GitHub Actions) and test their changes.
2
Nominate Services: Once verified in UAT, developers nominate their service on the 📋 Board tab — version is auto-captured from the live cluster. Add Jira IDs for the changes being released.
3
QA Validation: Run 🔀 Version Drift to check for mismatches, run 🤖 AI Readiness for health checks. Use 📜 Version History to rollback if needed.
4
Cutoff Time — Board Auto-Locks: At the configured cutoff time, the board auto-locks. No more regular nominations accepted. ⚠️ Exception Nominations are still possible with approver and reason (for late hotfixes).
5
Generate Artifacts: Go to 📦 Export tab → 🤖 AI Release Notes (fetches Jira ticket summaries via MCP). Export JSON/YAML manifests for production deploy pipelines.
6
Friday — Release Day: Deploy to production using exported manifests. Click ✅ Mark Released to archive the board. A new weekly cycle begins automatically.

🏁 Getting Started

🔐 Step 1: GitHub Login

Click "Login with GitHub" in the sidebar to authenticate. This enables deployment workflows and associates your identity with board actions.

Authentication modes:
  • OAuth 2.0 (recommended) — Secure browser-based login. Requires GITHUB_CLIENT_ID + GITHUB_CLIENT_SECRET.
  • PAT mode — Shared Personal Access Token for service accounts. Set GITHUB_TOKEN env var.
  • Supports GitHub Enterprise — set GITHUB_URL to your GHE instance.
Why it's needed:
  • Your GitHub identity is logged in the Audit Trail for every action
  • Required for the 🚀 Deploy tab (triggers GitHub Actions using your token)
  • Board nominations and AI features work without login — only Deploy requires auth

📋 Board & Nominations

📋 Release Board

The Board is the central hub for every release. It shows all services nominated for the upcoming Friday release with their locked versions.

How to nominate a service:
  1. Choose ● K8s Service or ⚡ Custom Component in the sidebar
  2. Select a service from the dropdown — Image Tag and Helm Chart are auto-filled from the live UAT cluster
  3. Enter your name in Your Name
  4. Enter associated Jira IDs (comma-separated, optional — e.g., BILL-101, BILL-102)
  5. Add optional Release Notes describing the change
  6. Click 🚀 Nominate
Key behaviors:
  • Version is locked at nomination time — if someone deploys a newer version to UAT later, the board keeps the original nominated version
  • To update, re-nominate the same service (previous version is saved in history)
  • Toggle between Tile view (▦) and List view (☰)
  • Each card shows: version tag, Helm chart, readiness badge, 🎫 Jira badges, nominator, date, and notes
  • Click 👁 Details to see full service info, 📜 History for version timeline, ✕ Remove to withdraw
Board states:
  • ● Open — Accepting nominations (Monday through Wednesday cutoff)
  • 🔒 Locked — Past cutoff. Regular nominations blocked. Exception nominations still allowed with reason + approver.
  • ✅ Released — Archived as read-only. A new cycle begins.
💡 Tip: The summary bar at the top shows nominated count, readiness breakdown (Ready / Review / Risk), release date, and cutoff time at a glance.

⚡ Custom Components (Non-K8s)

Not everything runs in Kubernetes. Custom Components let you nominate Spark jobs, PySpark pipelines, mobile apps, config rules, or any non-K8s artifact.

How it differs:
  • Click ⚡ Custom Component tab in the nomination form
  • Version is manually entered (not auto-detected from cluster)
  • Component types and descriptions are configured via CUSTOM_COMPONENTS env var
  • Jira IDs work the same — link Jira tickets to custom component releases
💡 Config example: CUSTOM_COMPONENTS=ingestion-pipeline:Spark:Main data ingestion,etl-transformer:PySpark:Data transformation

📜 Version History & Rollback

Every time a service is nominated or re-nominated, the previous version is saved in a timeline. Click the 📜 History button on any service card to view and rollback.

What you can do:
  • View full timeline — who nominated which version and when
  • Click "↩ Rollback to this version" to instantly revert to any previous nomination
  • All rollbacks are logged in the Audit Trail with from → to version
💡 Use case: You nominated billing-service v2.3.5, but QA found a regression. Rollback to v2.3.3 without redeploying — the board nomination reverts instantly.

⚠️ Exception Nominations (Post-Cutoff)

After the board is locked (at the configured cutoff time), developers can still nominate services as exceptions. This is for late hotfixes or critical changes that must go into the release. Every exception is tracked and reported for weekly review.

How it works:
  1. Try to nominate a service after the board is locked
  2. An ⚠️ Exception Nomination modal appears automatically
  3. Enter a Reason (why this is needed after cutoff) — required
  4. Enter an Approver (who approved this exception) — required
  5. Click Submit Exception Nomination
What gets tracked:
  • Board cards — Exception services show an ⚠️ EXCEPTION badge with approver and reason
  • Sidebar — Exception count displayed in Quick Info when > 0
  • Audit Trail — Logged as exception-nominate or exception-re-nominate with amber highlighting
  • AI Release Notes — A ⚠️ Post-Cutoff Changes section is added, flagging these as higher-risk items
  • Exception Report — GET /api/release/exceptions returns stats grouped by requester and approver
💡 Why it matters: Exception tracking creates accountability. Weekly reports show how many post-cutoff changes are happening and who is requesting them — helping the team reduce late changes over time.

🎫 Jira Integration

🎫 Linking Jira Tickets to Releases

Each nominated service can have Jira ticket IDs attached, connecting the release board to your team's issue tracker. This enables AI-powered release notes that summarize what actually changed based on Jira descriptions.

How to add Jira IDs:
  1. When nominating a service, enter Jira ticket IDs in the Jira IDs field
  2. Use comma separation for multiple tickets: BILL-101, BILL-102, BILL-103
  3. IDs must follow the standard format: PROJECT-NUMBER (e.g., DEVOPS-123)
  4. Jira IDs can be updated by re-nominating the service
Where Jira IDs appear:
  • Board tile view — Styled 🎫 badges on each service card
  • Board list view — Dedicated "Jira" column in the table
  • Service detail modal — Full Jira ID list when you click 👁 Details
  • AI Release Notes — "Jira Tickets" column + "What's Changed" section
  • AI Chatbot — Ask "What Jira tickets are in this release?" or about a specific service

🔌 Jira Connection — Dual-Mode Architecture

Release Readiness supports two methods for connecting to Jira, tried in order. If MCP fails or times out, the system automatically falls back to the REST API. Both methods are used for individual ticket lookups and fix version JQL searches.

Mode 1: MCP Server (recommended):
  • Connects to a Jira MCP (Model Context Protocol) server via JSON-RPC over HTTP
  • Individual tickets: calls jira_get_issue (or get_issue as fallback) for each Jira ID
  • Fix version search: auto-discovers search tools — tries search_jira_issues, jira_search_issues, search_issues, and jira_search until one succeeds
  • Quick-fail: 5-second timeout on MCP search — if MCP is unreachable, remaining tool names are skipped and the system falls back to REST API immediately
  • SSE support: handles both standard JSON responses and text/event-stream (Server-Sent Events) format from MCP servers
  • 3-layer response parsing: JSON → key-value text → raw text fallback — handles any MCP server's response format
Mode 2: Direct REST API (fallback):
  • Connects directly to your Jira instance at JIRA_BASE_URL using the standard REST API v2
  • Searches /rest/api/2/search with JQL: fixVersion = "PYY.MM.DD"
  • Tries both GET and POST — GET works for Jira Server/Data Center, POST for Jira Cloud
  • Returns full issue data: summary, description, type, status, priority, and components
Authentication:
  • MCP mode: sends Jira-Token header + Authorization: Bearer fallback + X-Jira-User-Email
  • REST API mode: prefers Bearer token (Jira Data Center PAT), falls back to Basic Auth (email:PAT for Jira Cloud)
  • SSL verification respects the SSL_VERIFY env var — can be disabled for self-signed certs
  • Proxy support via PROXY_URL — routes all Jira requests through your corporate proxy if configured
Configuration:
# Option 1: MCP Server
JIRA_MCP_URL=https://jira-mcp.your-org.com

# Option 2: Direct REST API
JIRA_BASE_URL=https://jira.company.com

# Authentication (used by both modes)
JIRA_EMAIL=jira-svc-account@company.com
JIRA_PAT_TOKEN=<your-jira-pat-token>
💡 Graceful degradation: If neither Jira MCP nor REST API is configured (or both are unreachable), release notes still generate — just without Jira descriptions. No errors, no breakage. The system logs detailed diagnostics to help troubleshoot connection issues.

🏷️ Fix Version Search & Component Mapping

When a Fix Version is set on the board (format: PYY.MM.DD), the system can automatically find all Jira tickets tagged with that fix version — not just the ones manually entered during nomination. This gives you the complete picture of what's in the release.

How it works:
  1. When AI Release Notes are generated, the system runs JQL: fixVersion = "P26.05.30"
  2. All matching tickets are fetched with their components field
  3. Tickets are auto-mapped to nominated services using component → service name matching
  4. Matching is case-insensitive and dash/underscore-normalized (e.g., billing-service matches component Billing_Service)
  5. Unmapped tickets (infrastructure, cross-cutting concerns) are listed separately in the release notes
Data sources combined:
  • Manual Jira IDs — entered during nomination (e.g., BILL-101, BILL-102)
  • Fix version search — auto-discovered tickets from fixVersion = "PYY.MM.DD" JQL
  • Both are merged — manual IDs take priority, fix version adds any missing tickets
💡 Tip: Set the Fix Version in the Export tab to match your Jira release version. This enables automatic ticket discovery — developers don't need to manually enter every Jira ID when nominating.

✅ Quality & Validation

🔀 Version Drift Detection

Compares the nominated version on the board against what's actually running in UAT right now. Catches cases where someone deployed a newer version after nomination.

Drift levels:
  • 🟢 No Drift — Nominated version matches what's live in UAT
  • 🟡 Minor Drift — UAT has a newer patch/minor version. Developer should re-nominate with the correct version.
  • 🔴 Major Drift — Different major version in UAT. Needs immediate investigation — possible unintended deployment.
💡 When to run: Run drift checks before cutoff (Wednesday) and after any UAT deployment to ensure the board reflects reality.

🤖 AI Readiness Check

Powered by Google Gemini. Performs automated health analysis on every nominated service by examining live Kubernetes cluster data (pods, events, resources, probes).

What AI checks:
  • Pod health — replica count, crash loops, restart count in last 48h
  • K8s events — warnings, errors, scheduling failures, ImagePullBackOff
  • Resources — CPU/memory requests vs limits, potential OOM risk
  • Probes — liveness/readiness probe configuration and failure rates
  • Image tag risk — flags :latest tags as unpinned supply chain risk
  • Version change risk — major version bumps flagged for extra review
Readiness scores:
  • 🟢 Ready (70–100) — Service is healthy. Safe to include in release.
  • 🟡 Review (40–69) — Some concerns found. Release Manager should investigate before approving.
  • 🔴 Risk (0–39) — Critical issues. Should not be released until resolved.
💡 Tip: Readiness results are shown as badges on each board card and in the summary bar at the top. Click 👁 Details on any service to see the full AI analysis.

📜 Audit Trail

Every action on the release board is recorded with timestamps, who did it, and what changed. Full traceability for compliance and post-mortems.

Events logged:
  • nominate — Service added to the board with version and Jira IDs
  • re-nominate — Version updated (from → to) with reason
  • exception-nominate — Post-cutoff nomination with approver and reason (amber highlight)
  • exception-re-nominate — Post-cutoff re-nomination with approver and reason
  • rollback — Version reverted via Version History
  • remove — Service withdrawn from the release
  • deploy_triggered — GitHub Actions workflow dispatched (includes run ID, who triggered, inputs)
  • finalize — Board locked at cutoff by Release Manager
  • unlock — Board manually unlocked for editing
  • release — Board marked as released (archived)
  • auto_release — System auto-released the board after release date passed without manual release
  • auto_archive — System auto-archived the board and started a new cycle

📦 Export & Release Notes

📦 Export Manifests

Download the release manifest in JSON or YAML format. These machine-readable manifests can be fed into Helm, ArgoCD, or custom deployment pipelines for the production release.

Manifest contents:
  • Release name, date, status, cutoff timestamp, fix_version (PYY.MM.DD)
  • Per-service: name, image, image_tag, helm_chart, nominated_by, jira_ids, readiness, notes

🤖 AI Release Notes (Jira-Enriched)

Generates professional, human-readable release notes powered by Google Gemini. When services have Jira IDs, the system fetches ticket details via the Jira MCP server and weaves them into the notes.

What gets generated:
  1. Executive Summary — 2-3 sentence overview of the release
  2. Service Table — Service | Version | Jira Tickets | Change Summary | Risk Level
  3. What's Changed — Changes organized by type:
    • 🆕 New Features — from Story/Feature Jira tickets
    • 🐛 Bug Fixes — from Bug tickets
    • ⚡ Improvements — from Improvement tickets
    • 🔧 Maintenance — from Task tickets
  4. AI Risk Assessment — Overall release risk based on readiness scores and change scope
Output options:
  • 📋 Copy Markdown — paste directly into Jira, Confluence, or Teams
  • 📄 Copy as HTML — paste into email clients
✨ Gemini Powered badge appears when notes are generated by AI. 📄 Template badge appears when Gemini is not configured (deterministic fallback). Both modes include Jira data when available.

🔒 Board Lifecycle Controls

  • 🔒 Lock Board (Cutoff) — Freezes nominations. The board auto-locks at the configured cutoff time. Can also be manually locked by a Release Manager. All regular nomination buttons are disabled after locking.
  • 🔓 Unlock Board — Reopens a locked board for editing. Use this if you need to add/modify nominations after cutoff without going through the exception flow.
  • ✅ Mark Released — Archives the board to release history. A new cycle is created for the next Friday.
🔄 Auto-Release (Smart Cycle Management):
  • If the release date passes and nobody clicked "Mark Released", the system auto-releases the board on the next day (midnight UTC).
  • Auto-released boards are tagged with released_by: "system (auto-released)" in the audit trail.
  • The old board is archived to history and a fresh board is created automatically — no manual intervention needed.
  • This prevents stale boards from blocking the next week's release cycle.
🕐 Smart Cutoff Protection:
  • If the board was created in a previous week, the system recalculates the cutoff for the current release window instead of using the stale stored value.
  • This prevents the board from incorrectly showing "Locked" due to an outdated cutoff time.
  • Same protection applies to the nomination endpoint — nominations are checked against the effective cutoff, not the stored one.
💡 Full lifecycle: Open → Locked (at cutoff) → Released (manual or auto) → Archived → New Board. The system ensures a new cycle always starts, even if the Release Manager forgets to click "Mark Released".

📚 Release History

🏷️ Fix Version Selector:
  • Use the dropdown at the top to select a specific past release by its fix version (e.g. P26.05.30)
  • Each option shows: fix_version · release_date (N services)
  • Selecting a release renders a full detail table showing: #, Service, Version, Helm Chart, Readiness, Nominated By, Jira IDs, and Notes
  • A summary badge shows service count and how many were marked "Ready"
📋 Copy to Clipboard:
  • Click the 📋 Copy button to export the selected release's table as tab-separated text
  • Paste directly into Excel, Google Sheets, or email — columns are perfectly aligned
  • Includes fix version header and release metadata
Browse All Releases:
  • Below the selector, collapsible cards show all archived releases
  • Click any card to expand and see the full service table
  • Shows release date, fix version, service count, exception count, who released it, and when
💡 Tip: Auto-released boards (when the RM forgot to click "Mark Released") also appear in history, tagged as released_by: system (auto-released) so you can tell them apart from manual releases.

🖥️ Environments & Bi-Directional Cluster Support

🔄 Bi-Directional Cluster Architecture (DEPLOY_ENV)

The dashboard supports deployment in both UAT and Production clusters. The DEPLOY_ENV environment variable controls which cluster is "local" (in-cluster) and which is "remote" (connected via API token).

DEPLOY_ENV UAT Env Tab Prod Env Tab
uat (default) Reads LOCAL cluster Connects REMOTELY via PROD_CLUSTER_*
prod Connects REMOTELY via UAT_CLUSTER_* Reads LOCAL cluster
💡 Why bi-directional? Firewalled environments (e.g. UAT cannot reach Prod network). Deploy the app in both clusters — each reads its own local services and connects remotely to the other.

🖥️ UAT Environment

A live view of all services currently deployed in the UAT Kubernetes namespace. Independent of the nomination board — shows what's actually running right now.

What it shows:
  • Service name, kind (Deployment / StatefulSet / DaemonSet), replica status (ready/desired)
  • Current image tag and Helm chart version from the cluster
  • Availability status — whether all pods are ready
  • Board Status column — shows ✅ Nominated (matching version), ⚠️ Drift (version mismatch), or — (not nominated)
  • Summary bar: total services, namespace, healthy count, nominated count with drift warnings
  • Click 🔄 Refresh to get the latest data from the cluster
Connectivity:
  • DEPLOY_ENV=uat → reads local in-cluster K8s API (default)
  • DEPLOY_ENV=prod → connects remotely via UAT_CLUSTER_API + UAT_CLUSTER_TOKEN

🏭 Prod Environment

Lists live production services and cross-references with the board to show which prod components are nominated.

What it shows:
  • All Deployments, StatefulSets, and DaemonSets from the production namespace
  • Board Status — ✅ Nominated or Not Nominated (no version drift — prod and UAT versions differ by design)
  • Summary bar: services count, namespace, cluster URL, nominated vs not-nominated counts
Connectivity:
  • DEPLOY_ENV=prod → reads local in-cluster K8s API
  • DEPLOY_ENV=uat → connects remotely via PROD_CLUSTER_API + PROD_CLUSTER_TOKEN
Export Not Nominated:
  • 📋 Show Not Nominated — inline table of all prod services not on the board
  • 📥 Export CSV — download for spreadsheet tracking
  • 📥 Export JSON — machine-readable format for deployment workflow input
💡 Setup: Create a read-only ServiceAccount on the remote cluster:
oc create sa release-readiness-reader -n <ns> then oc adm policy add-role-to-user view system:serviceaccount:<ns>:release-readiness-reader -n <ns>

🚀 Deploy (GitHub Actions)

Trigger GitHub Actions workflows directly from the dashboard. Connects to your org's deployment repository and auto-discovers all available workflows.

How it works:
  1. Login with GitHub first (OAuth required for API access)
  2. Workflows are auto-discovered from the repo — just add workflow YAML files and they appear
  3. Each workflow card shows: name, file, last run result (✅/❌), duration, who ran it
  4. Click a workflow to expand — dispatch inputs are parsed from the YAML and rendered as form fields
  5. Fill in inputs (service name, version, namespace) and click ▶️ Run Workflow
  6. Real-time status polling shows step-by-step progress with live updates
Safety features:
  • environment input is 🔒 forced to UAT — staging/production cannot be targeted from this tool
  • Every deploy trigger is logged in the Audit Trail with GitHub user, inputs, and run ID
  • Deploy history shows all past runs with status and links to GitHub
Configuration:
DEPLOY_REPO=your-org/app-deployments
GITHUB_CLIENT_ID=<oauth-client-id>
GITHUB_CLIENT_SECRET=<oauth-secret>
BASE_URL=https://release-readiness.your-domain.com
💡 Tip: Workflows need a workflow_dispatch trigger with inputs. choice inputs render as dropdowns, string inputs as text fields.

🧪 QA Pipeline (Quality Gate)

🧪 Overview

The QA tab provides a structured 5-stage quality gate pipeline that takes your release from a locked board all the way to production-ready. It uses a GitOps approach — pushing version.yaml manifests to specific branches in your deployment repo, which ArgoCD (or your CD tool) can auto-deploy from.

🔄 Flow: Board Lock → Prepare E2E → QA Status → Run Tests → Drift Check → Push to Prod/PreProd
Prerequisites:
  • GitHub authentication (OAuth or PAT)
  • QA_DEPLOY_REPO — the GitHub repo where version.yaml is pushed (e.g. org/app-deployment)
  • QA_TEST_REPO — the GitHub repo containing your test pipeline workflows
  • The release board must be locked (manually or via cutoff time)

1 Prepare E2E Environment

Generates a version.yaml manifest from the locked release board and pushes it to the e2e branch of your QA_DEPLOY_REPO. If ArgoCD is watching this branch, it auto-deploys to the E2E environment.

What the manifest contains:
  • Nominated services from the board (with their locked image tags)
  • Non-nominated production services — carried forward from the current prod cluster to form a complete deployment
  • Metadata: release date, QA namespace, generation timestamp
💡 Tip: After clicking Prepare E2E, a version preview table shows all services with their source (Board vs Production). You'll also get a direct link to the commit on GitHub.

2 QA Namespace Status

Displays live services running in the QA/UAT Kubernetes namespace (configured via QA_NAMESPACE, defaults to uat-testing). Use this to verify that ArgoCD has synced the E2E deployment and all pods are healthy.

What you see:
  • Service name, running image tag, kind (Deployment/StatefulSet), and pod status
  • Ready replicas vs desired replicas
  • Click 🔄 Refresh to poll the cluster for the latest state

3 Test Pipelines

Triggers test suites against the QA environment via GitHub Actions workflow_dispatch. Tests run in your QA_TEST_REPO and are configurable per suite.

Available test suites:
  • 🧪 Smoke Tests — Quick health checks (workflow: QA_TEST_SMOKE_WORKFLOW)
  • 🧪 E2E Tests — Full end-to-end scenario tests (workflow: QA_TEST_E2E_WORKFLOW)
  • 🧪 Regression Tests — Comprehensive regression suite (workflow: QA_TEST_REGRESSION_WORKFLOW)
  • ▶▶ Run All — Triggers all three suites in parallel
💡 Tip: Each test shows a status card (Pass ✅ / Fail ❌ / Running ⏳) and a link to view the full workflow run on GitHub. You can run individual tests or all at once.

4 Version Drift Check

Compares the current board nominations + production service versions against the version.yaml that was pushed to the e2e branch. This catches any changes made after the initial E2E preparation — for example, late exception nominations or production hotfixes that happened since the E2E push.

Drift types detected:
  • Tag Changed — A service's image tag differs from what was pushed to E2E
  • New Service — A service was added to the board after E2E was prepared
  • Removed — A service that was in E2E is no longer on the board
⚠️ If drift is detected: Consider re-running Step 1 (Prepare E2E) to push an updated version.yaml, then re-run your test suites before proceeding to production.

5 Prepare Prod / PreProd

The final stage. Generates the production-ready version.yaml (including a change ticket reference) and pushes it to both the prod and preprod branches of your QA_DEPLOY_REPO.

What happens:
  • Enter your Change Ticket number (e.g. CHG0012345) — this is embedded in the manifest for traceability
  • The version.yaml is pushed to both prod and preprod branches simultaneously
  • ArgoCD (or your CD tool) watching these branches will begin the production rollout
  • Commit links are displayed for both branches so you can verify the push on GitHub
✅ After this step: Your release is deployed to production. Return to the 📋 Board tab and click ✅ Mark Released to archive the board and start the next release cycle.

🔌 Confluence MCP Server Connection

Connects to your organization's Confluence MCP (Model Context Protocol) server to search pages, retrieve content, and query by labels. Falls back to the Confluence REST API if no MCP server is configured.

Dual-mode architecture:
  • Option 1: MCP Server (recommended) — JSON-RPC over HTTP, same protocol as Jira MCP. Set CONFLUENCE_MCP_URL.
  • Option 2: REST API (fallback) — Direct Confluence Cloud/Server REST API. Set CONFLUENCE_BASE_URL + auth credentials.
  • Both modes support AI summarization via Gemini when GEMINI_API_KEY or GCP_PROJECT_ID is set.
Configuration:
# Option 1: MCP Server
CONFLUENCE_MCP_URL=https://confluence-mcp.your-org.com

# Option 2: Direct REST API
CONFLUENCE_BASE_URL=https://your-org.atlassian.net/wiki

# Authentication (shared with Jira if same Atlassian instance)
CONFLUENCE_EMAIL=your-email@company.com
CONFLUENCE_PAT_TOKEN=<api-token>

# Optional: restrict to specific spaces
CONFLUENCE_DEFAULT_SPACES=DEV,OPS,REL
💡 Shared credentials: If your Jira and Confluence are on the same Atlassian Cloud instance, the system automatically reuses JIRA_EMAIL and JIRA_PAT_TOKEN for Confluence auth — no extra configuration needed.

❓ Frequently Asked Questions

Q: What happens if I nominate the same service twice?
The old version is saved in Version History and the board updates to the new version. Both nominations are logged in the Audit Trail.
Q: Can I add Jira IDs after initial nomination?
Yes — re-nominate the service with the same version and add the Jira IDs. The field merges with existing IDs.
Q: What if Jira MCP server is not configured?
Everything works normally. Jira IDs are still displayed on the board, but the AI Release Notes won't include Jira ticket descriptions — they'll use service metadata and notes instead.
Q: Can I deploy to production from this tool?
No. The environment input is locked to UAT. Production deployments must go through your standard CI/CD pipeline using the exported manifests.
Q: How does the board persist data?
Board state is stored in a Kubernetes ConfigMap in the same namespace. If the service account lacks ConfigMap write permissions, data falls back to a file on a PersistentVolume. Both methods are automatic — no configuration needed. In mock mode, data is in-memory (resets on restart).
Q: Can I nominate a service after the board is locked?
Yes — as an Exception Nomination. You'll need to provide a reason and an approver name. The service will be flagged with an ⚠️ badge and included in the weekly exception report.
Q: Where can I see how many exceptions happened this week?
The sidebar shows exception count when > 0. For detailed stats, the Release Manager can call GET /api/release/exceptions which returns counts grouped by requester and approver.
Q: What if Gemini is not configured?
AI Readiness and AI Chatbot are disabled. Release Notes fall back to a deterministic template (still includes Jira IDs if provided). The board and all other features work without AI.
Q: What is DEPLOY_ENV and how does bi-directional mode work?
Set DEPLOY_ENV=uat (default) when deploying to the UAT cluster, or DEPLOY_ENV=prod when deploying to the Production cluster. The "local" tab reads from the in-cluster K8s API, while the "remote" tab connects via PROD_CLUSTER_* or UAT_CLUSTER_* env vars. This enables full functionality even when clusters are firewalled from each other.
Q: How does the remote cluster tab connect?
It creates a separate Kubernetes API client using the relevant *_CLUSTER_API and *_CLUSTER_TOKEN env vars. The ServiceAccount only needs view role (read-only). No write access to the remote cluster.
Q: What is the Fix Version format?
Auto-generated as PYY.MM.DD from the release date (e.g., P26.05.08). It can be manually edited in the Export tab. Used to aggregate Jira tickets via JQL for AI Release Notes.
Q: Are Jira IDs required when nominating?
No — Jira IDs are optional. You can nominate services without them. If a Fix Version is set, AI Release Notes will aggregate Jira tickets from the fix version automatically.
Q: What does the Confluence Agent do?
It searches your org's Confluence wiki using natural language and generates AI-summarized answers. Think of it as a smart search that reads runbooks, architecture docs, and procedures for you — no need to leave the dashboard. Supports both MCP server and direct REST API modes.
Q: Do I need a separate Confluence token?
Not if your Jira and Confluence are on the same Atlassian Cloud instance. The system automatically reuses JIRA_EMAIL and JIRA_PAT_TOKEN for Confluence. Set CONFLUENCE_EMAIL and CONFLUENCE_PAT_TOKEN only if they're different.
Q: What happens if I forget to click "Mark Released"?
The system auto-releases the board after midnight UTC on the day following the release date. The board is archived to history with released_by: "system (auto-released)" and a fresh board is created for the next cycle. No manual intervention is needed.
Q: How do I view past release details in the History tab?
Go to the 📚 History tab and use the Fix Version dropdown to select a past release. A full detail table will render showing all nominated services with versions, readiness, Jira IDs, and notes. Click 📋 Copy to export the table for pasting into Excel or email.
