📖 How Release Readiness Works

Release Readiness is an AI-powered release coordination board. Instead of posting versions in Teams and compiling spreadsheets, developers nominate services here and AI validates everything before release. Deployments to UAT are triggered via GitHub Actions workflows from the Deploy tab.

🔐 GitHub Login

Release Readiness uses GitHub OAuth for authentication. Click "Login with GitHub" in the sidebar to connect your account. This is required for triggering deployments from the Deploy tab.

Details:
  • Secure OAuth 2.0 flow — no passwords stored in the app
  • Your GitHub identity is used for audit trail (who triggered which deploy)
  • Requires read access to the deployment repo and Actions write access to trigger workflows
  • Also supports PAT (Personal Access Token) mode — set GITHUB_TOKEN env var for service accounts

📋 Board

The Board is the central hub of the release cycle. It shows all services nominated for the upcoming release. Supports both K8s services (auto-detected) and Custom Components (manual entry).

How it works:
  • Select a service from the dropdown — the image tag and Helm version are auto-filled from the live UAT cluster
  • The version is locked at nomination time — if someone deploys a newer version to UAT later, the board still shows the original nominated version
  • To update, re-nominate the service (logged in audit trail) or use 📜 Version History to rollback
  • Toggle between Tile view (▦) and List view (☰)
💡 Tip: The board has 3 states — Open (accepting nominations), Locked (past cutoff, no changes), and Released (read-only archive).

📜 Version History

Every time a service is nominated or re-nominated, the previous version is saved. Click the 📜 button on any service card to view all past nominations.

What you can do:
  • View a timeline of all nominated versions with who/when
  • Click "↩ Rollback to this version" to revert to any previously nominated version
  • Rollbacks are logged in the audit trail for traceability
💡 Use case: You nominated v2.3.5, but QA found a bug. Rollback to the previously nominated v2.3.3 without needing to redeploy.

🔀 Version Drift

Compares the nominated version on the board against what's actually running in UAT right now.

Drift levels:
  • 🟢 No Drift — Nominated version matches UAT
  • 🟡 Minor Drift — UAT has a newer patch/minor version. Developer should re-nominate.
  • 🔴 Major Drift — Different major version. Needs investigation.

🤖 AI Readiness

Uses Google Gemini to perform automated health checks on every nominated service. Analyzes live Kubernetes data to determine release safety.

What AI checks:
  • Pod health — replicas, crash loops, restarts
  • K8s events — warnings, errors, scheduling issues
  • Resources — CPU/memory usage within limits
  • Container status — OOMKills, image pull errors, probe failures
  • Version risk — major version bumps flagged for review
Readiness scores:
  • 🟢 Ready (70-100) — safe to release
  • 🟡 Review (40-69) — release manager should review
  • 🔴 Risk (0-39) — should not release until resolved

📜 Audit Trail

Records every action on the release board with timestamps, who performed it, and what changed.

What gets logged:
  • Nominations — who nominated which service and when
  • Re-nominations — version changes with from → to versions
  • Rollbacks — when a previous version is restored via Version History
  • Removals — who removed a service
  • Deployments — who triggered which workflow + GitHub run ID
  • Board Lock / Release Complete — lifecycle events

📦 Export

Generate release artifacts and control the board lifecycle.

Available actions:
  • Export JSON / YAML — download the release manifest for Helm/ArgoCD workflows
  • AI Release Notes — Gemini generates human-readable release notes with change summaries and risk flags
  • Lock Board (Cutoff) — freezes the board. No more nominations.
  • Mark Released — archives the board. A new cycle begins.

🖥️ UAT Env

Shows all services currently deployed in the UAT Kubernetes namespace — a live view of what's running, independent of the nomination board.

What it shows:
  • Service name, kind (Deployment/StatefulSet), replicas status
  • Current image tag and Helm chart version
  • Use this to verify what's running vs what's nominated on the board

🚀 Deploy

Connects to your GitHub deployment repository (set by DEPLOY_REPO env var, e.g., your-org/app-deployments) and lists all active GitHub Actions workflows.

How it works:
  • Workflows are auto-discovered — no manual config needed. Add a new .yml workflow to your repo and it appears here.
  • Each card shows: workflow name, file, last run status (✅/❌), duration, and who ran it
  • Click a card to expand it — dispatch inputs are parsed from the YAML and rendered as form fields
  • Fill in inputs (service, image version) and click ▶️ Run Workflow
  • environment input is 🔒 locked to UAT — staging/production cannot be selected from this tool
  • Real-time status polling shows step-by-step progress after triggering
Configuration:
DEPLOY_REPO=your-org/app-deployments
GITHUB_CLIENT_ID=<oauth-client-id>
GITHUB_CLIENT_SECRET=<oauth-secret>
BASE_URL=https://release-readiness.your-domain.com
💡 Tip: Your workflows must have a workflow_dispatch trigger with inputs defined. The Deploy tab renders dropdowns for choice inputs and text fields for string inputs.

💬 AI Chatbot

A conversational AI assistant powered by Google Gemini. Ask natural language questions about the release and get instant, data-backed answers with formatted tables and status reports.

What you can ask:
  • "What's in this release?" — lists all nominated services with versions
  • "Check drift" — compares nominated vs live versions
  • "Show readiness scores" — AI health check results for all services
  • "Show audit trail" — recent board activity with who/what/when
  • "What's running in UAT?" — live cluster services overview
  • "Is billing-service included?" — status of a specific service
Features:
  • Multi-turn conversations with session memory
  • Quick-ask suggestion pills for common queries
  • Markdown rendering — tables, bold, code, lists
  • Clear button resets conversation history
💡 Tip: The chatbot uses Gemini function-calling to query live board data. Requires GCP_PROJECT_ID or GEMINI_API_KEY configured. In mock mode, contextual keyword-matched responses are returned.

🔄 Typical Release Workflow

1️⃣ Monday–Wednesday: Developers deploy to UAT via the 🚀 Deploy tab, then nominate confirmed versions on the Board
2️⃣ Before cutoff: QA/Release Manager runs Version Drift + AI Readiness checks
3️⃣ Need to change? Use 📜 Version History to rollback, or re-nominate with a new version
4️⃣ Wednesday 5 PM: Board is locked (cutoff). No more nominations.
5️⃣ Thursday: Export manifests, generate AI release notes, update Jira
6️⃣ Friday: Deploy to production. Mark Released. New cycle begins.
